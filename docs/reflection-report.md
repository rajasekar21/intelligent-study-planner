# Reflection Report – AI Powered Study Planner

## 1) Which AI tools were used and how

For this assignment I used **Cursor AI** throughout development as my main coding assistant. I used it in an iterative way, not in a copy-paste way. My process was:

1. I decided the exact feature/fix I wanted.
2. I asked Cursor for a draft implementation.
3. I tested the output in the running app.
4. I manually edited and corrected the code before committing.

I used Cursor for backend APIs, frontend wiring, styling improvements, refactoring, and documentation drafting. I still treated myself as responsible for correctness.

## 2) Example prompts used during development

Some real examples of the type of prompts I used:

- "Add JWT auth middleware and protect all non-auth routes in FastAPI."
- "Add complete topic CRUD with search in backend and React UI."
- "Refactor App.jsx into route-based pages and reusable components."
- "Create AI usage log table in DB and expose create/list/delete/export APIs."
- "Migrate from monolithic AI utility (`ai.py`) to standalone AI microservice."
- "Generate Playwright script for end-to-end workflow recording."
- "Improve frontend look but keep it simple and assignment-friendly."

The outputs were useful, but usually first drafts. I almost always changed them after testing.

## 3) Parts generated by AI vs manually coded

### Mostly AI-assisted

- Initial versions of endpoint wiring in `backend/app/main.py`
- Validation schema additions in `backend/app/schemas.py`
- Frontend request integration in `frontend/src/api.js`
- Early UI structure suggestions for `frontend/src/App.jsx`
- Draft docs and markdown structure in `docs/*`

### Mostly manual by me

- Final auth flow hardening in `backend/app/auth.py`
- Access control corrections and role constraints in `backend/app/main.py`
- Refactor into `frontend/src/pages/*` and `frontend/src/components/*`
- SQLite setup and runtime fixes (`backend/app/database.py`, env setup)
- Integration fixes for CORS/ports and microservice communication
- Final wording and cleanup of README/reflection docs

Even in files where AI generated code, I manually reviewed and rewrote sections. For example, the old `backend/app/ai.py` logic was later moved into `ai-service/main.py` with manual integration work in backend using `httpx`.

## 4) Did AI help or hinder understanding?

Honestly, it did both.

### Where AI helped

- It saved time in repetitive coding tasks.
- It helped me move faster from idea to working draft.
- It gave good scaffolding for endpoints, schemas, and UI wiring.
- It made refactoring less intimidating because I could request structural changes in steps.

### Where AI hindered (or needed correction)

- Sometimes outputs were generic and not aligned with my current project state.
- A few suggestions looked correct but broke integration when tested.
- Some generated code added unnecessary complexity for assignment scope.

So AI improved speed, but only when I actively reviewed it. If I had accepted everything blindly, I would have submitted unstable code.

## 5) Issues that arose integrating AI-generated code

Key issues I faced while integrating AI output:

1. **CORS mismatch issue**  
   Frontend was running on `5174` but backend initially allowed only `5173`, so login failed with fetch errors.

2. **Auth/doc mismatch**  
   JWT middleware had been implemented in code, but README still had an outdated TODO line saying it was pending.

3. **Dependency compatibility issue**  
   Password hashing setup hit bcrypt/passlib compatibility problems during sample user initialization.

4. **Automation script fragility**  
   First Playwright script used selectors that were too brittle and failed when UI structure changed.

5. **Over-generated artifacts**  
   Multiple generated demo videos got committed temporarily; I had to clean and keep only the final required one.

These were not theoretical issues; they required debugging and real corrections.

## 6) What I learned from debugging AI output

The debugging phase was the most educational part of this project.

- I learned to verify each layer independently: backend API, frontend calls, DB persistence, and docs consistency.
- I got better at FastAPI dependency-based auth design (`get_current_user`, role checks).
- I understood practical integration details better (CORS, env config, service URLs, port mismatches).
- I learned that prompt quality matters: constraints like "keep it minimal", "no overengineering", and "preserve behavior" gave much better results.
- I learned to use AI as a collaborator, not an authority.

Overall, Cursor AI definitely improved productivity, but my understanding improved mostly when I debugged and corrected AI-generated code myself. That manual debugging step is what made the project submission-ready.
